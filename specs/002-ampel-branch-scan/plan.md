# Implementation Plan: AMPEL Branch Protection Scanning Plugin

**Branch**: `002-ampel-branch-scan` | **Date**: 2026-02-11
**Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/002-ampel-branch-scan/spec.md`

## Summary

Build a complyctl plugin that matches assessment plan rules
against granular AMPEL policy files and merges them into a
combined bundle (Generate), then invokes the AMPEL toolchain
(snappy, ampel) to scan branch protection rules on GitHub
and GitLab repositories (Scan). The plugin implements the
complyctl scanning provider interface
(`pkg/plugin.Plugin`), isolates the rule-to-AMPEL
conversion layer, handles DSSE-wrapped attestation output,
and groups multi-target results into proper observations.

**Structure Decision**: Single binary plugin following the
openscap-plugin layout. Eight packages (config, server, convert,
toolcheck, targets, scan, results, intoto) provide granular separation
per the user's requirement. Each package has a single, clearly
defined responsibility.

### Package Responsibilities

| Package | Responsibility | Key Dependency |
|---------|---------------|----------------|
| `main` | Plugin registration via `plugin.Serve()` | pkg/plugin |
| `config` | Workspace path helpers, directory creation | stdlib + hclog |
| `server` | Implement `plugin.Plugin` (Generate, Scan, Describe), orchestrate other packages | pkg/plugin |
| `convert` | Match assessment requirement IDs to granular AMPEL policies, merge into bundle | encoding/json |
| `toolcheck` | Validate snappy, ampel are on PATH | os/exec |
| `targets` | Parse and validate repository URLs, detect platform | stdlib |
| `scan` | Invoke snappy + ampel verify per repo, capture output | os/exec |
| `results` | Parse ampel verify output → ScanResponse, write per-repo JSON | encoding/json |
| `intoto` | In-toto statement and DSSE envelope types for attestation parsing | encoding/json |

### API Isolation for Gemara Migration

The plugin implements the Gemara-native scanning provider
interface, replacing the former C2P (`compliance-to-policy-go/v2`)
`policy.Provider` approach:

**`server` package** — Implements `pkg/plugin.Plugin`
(Generate, Scan, Describe RPCs). Configuration is stateless;
Generate receives `[]plugin.AssessmentConfiguration` and global
variables; Scan receives targets with per-target variables;
Describe reports health and required variables.

**`convert` package** — The only package that understands the
rule-to-AMPEL mapping:
1. `MatchPolicies` accepts assessment requirement IDs and
   matches them against granular policy file IDs
2. `LoadGranularPolicies` and `MergeToBundle` are format-only
   and independent of the upstream policy model
3. All downstream packages (scan, results, targets, config)
   remain untouched

**`results` package** — `ToScanResponse` maps `[]*PerRepoResult`
to `*plugin.ScanResponse`, grouping findings by CheckID into
assessment log entries with multiple subjects per check.

### Test Architecture

**Unit tests per package** cover:
- Happy path with valid inputs
- Unexpected/invalid inputs (empty, nil, malformed)
- Boundary conditions (empty policy, empty target list)
- Error propagation

**Mock fixtures** in `convert/testdata/`:
- Mocked assessment configuration (full and subset variants)
- Expected AMPEL policy output for each plan variant
- Pre-existing broader AMPEL policy for scope-honoring tests

**Verification of assessment plan ↔ AMPEL policy linkage**:
- Table-driven tests compare generated policy against expected
  fixtures
- Each test case modifies the input assessment plan and verifies
  the output changes accordingly
- Tests assert exact tenet count, tenet IDs, and CEL expression
  content
