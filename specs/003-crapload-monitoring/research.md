# Research: CRAP Load Monitoring

## R1: Gaze Tool Capabilities and Integration

**Decision**: Use `gaze crap` command with `--format=json` for machine-readable output in CI, and threshold flags (`--max-crapload`, `--max-gaze-crapload`, `--crap-threshold`, `--gaze-crap-threshold`) for enforcement.

**Rationale**: Gaze provides both CRAP and GazeCRAP computation natively. The `--format=json` output conforms to an embedded schema (retrievable via `gaze schema`). Threshold enforcement is built into the tool via exit codes (non-zero on violation), eliminating the need for custom comparison logic.

**Alternatives considered**:
- Custom script to compute CRAP from coverage data: Rejected because Gaze already handles this, aligning with Constitution Principle V (Do Not Reinvent the Wheel).
- SonarQube CRAP integration: Rejected because it does not support GazeCRAP (contract coverage), which is a core requirement.

## R2: Gaze Installation in CI

**Decision**: Install Gaze via `go install github.com/unbound-force/gaze/cmd/gaze@latest` in CI workflows, consuming the tool directly from its repository.

**Rationale**: The user explicitly requested consuming gaze directly from its repository. `go install` is the most straightforward approach for a Go project already using Go in CI. This avoids adding Homebrew as a CI dependency.

**Alternatives considered**:
- Homebrew install in CI: Rejected because it adds unnecessary dependency and setup time.
- Pre-built binary download from GitHub releases: Viable alternative if `go install` proves too slow, but the Go toolchain is already available in the CI environment.
- Vendoring gaze: Rejected as it would require maintaining the dependency.

## R3: Baseline Threshold Storage

**Decision**: Store baseline thresholds as a JSON file committed to the repository at `.gaze/baseline.json`. The baseline is generated by running `gaze crap --format=json ./...` against the main branch and extracting per-function scores.

**Rationale**: A committed file ensures baselines are versioned alongside code, reviewable in PRs, and accessible to CI without external storage. JSON format enables programmatic comparison.

**Alternatives considered**:
- CI artifact storage: Rejected because baselines would not be versioned with code and would be harder to inspect.
- Database or external service: Over-engineered for this use case.
- Aggregate-only thresholds (single number for entire codebase): Rejected because per-function thresholds provide more precise regression detection.

## R4: PR Comment Strategy

**Decision**: Use GitHub Actions' `github-script` or `gh` CLI to post/update a single PR comment with CRAP analysis results. The comment will be identified by a unique marker to enable updating rather than creating duplicate comments on re-runs.

**Rationale**: A single, updateable comment keeps the PR conversation clean. Using built-in GitHub Actions capabilities avoids external dependencies.

**Alternatives considered**:
- Third-party actions for PR comments (e.g., `marocchino/sticky-pull-request-comment`): Viable but adds an external dependency.
- Check run annotations: Provides inline feedback but does not offer the summary view that a PR comment provides. Could be added later.

## R5: Reusable Workflow Design

**Decision**: Create a reusable workflow (`workflow_call`) with configurable inputs having reasonable defaults. Inputs include: Go version, gaze version, CRAP threshold, GazeCRAP threshold, max crapload, max gaze crapload, baseline file path, and coverage profile path.

**Rationale**: The user explicitly requested flexibility with reasonable defaults for future centralisation in `org-infra`. This aligns with Constitution Principle VII (Convention Over Configuration) — defaults cover the common case, inputs allow deviation.

**Alternatives considered**:
- Composite action: Less flexible than reusable workflows for multi-step processes requiring different runners or permissions.
- Hardcoded values: Rejected per user requirement for flexibility and future org-infra centralisation.

## R6: Event-Based Metrics Publishing Strategy

**Decision**: Create a dedicated `ci_crapload_main.yml` workflow with `push` (to main, paths-filtered for Go files) and `workflow_dispatch` triggers. Metrics are published to GitHub Pages for Grafana consumption. Event-based triggers avoid wasted CI runs when no Go code has changed.

**Rationale**: Event-based publishing only runs when Go code is merged, avoiding unnecessary scheduled runs. Separate workflow keeps concerns isolated per Constitution Principle II (Simplicity & Isolation). GitHub Pages provides a free, zero-dependency hosting endpoint for Grafana's Infinity datasource.

**Alternatives considered**:
- Scheduled daily cron job: Rejected because it runs even when no code has changed, wasting CI resources.
- Adding to existing `ci_scheduled.yml`: Would mix concerns (security scanning vs quality metrics).
- External cron job: Over-engineered and introduces infrastructure outside the repository.
- GitHub Gist as data store: Viable but less "official" than GitHub Pages and has 100MB limit.

## R7: Local Testing Experience

**Decision**: Provide a Makefile target (`make crapload`) that runs the same analysis locally, and a `quickstart.md` with step-by-step instructions. The Makefile target will install gaze if not present, generate coverage, and run the analysis.

**Rationale**: The user explicitly requested it should be "super easy for a human to test the implementation locally." A Makefile target aligns with Constitution requirement ("Repositories MUST use a Makefile to centralize code-specific commands") and provides a single command to reproduce CI behaviour locally.

**Alternatives considered**:
- Shell script only: Would not align with Constitution's Makefile requirement.
- Docker-based local testing: Over-engineered for a Go tool that installs trivially.
