**Feature Branch**: `001-gemara-native-workflow`
**Created**: 2026-02-12
**Status**: Draft
**Input**: User description: "Update complyctl to decoupled workflow where Gemara policy authoring is centralized (with help from Gemara MCP server), while complyctl provides a lightweight, drop-in runtime for admins to pull in policies and execute scans."

## Clarifications

### Session 2026-03-02

- Q: What validation should be applied to user-provided policy URLs in `complyctl init`? в†’ A: OCI reference format validation (registry/repo:tag or @digest). Catches injection-like inputs and malformed URLs without being overly restrictive. Applies to both `init` interactive prompts and `config.go` policy entry parsing. (PR #381 feedback вЂ” marcusburghardt)
- Q: How should duplicate policies be handled during `complyctl init`? в†’ A: Warn and skip. Log the duplicate, continue execution. A duplicate policy doesn't corrupt state or change outcome вЂ” erroring forces unnecessary restart of interactive flow. (PR #381 feedback вЂ” marcusburghardt)
- Q: What should `complyctl init` use as the default policies list? в†’ A: Empty list with commented-out example showing `url` + `id` structure. No fake URL in actual config вЂ” comments show the format without polluting real data. Consistent structural guidance without a footgun. (PR #381 feedback вЂ” marcusburghardt)
- Q: How should the OpenSCAP provider surface the `oscap` command when it hangs or times out? в†’ A: Two changes. (1) Promote command logging from `hclog.Debug` to `hclog.Info` in `executeCommand()` so the full command appears in `.complytime/complyctl.log` вЂ” go-plugin filters Debug but passes provider Info through. Format as a shell-ready string (`strings.Join`), not a Go slice. (2) On context timeout, include the command in the error message so the admin can copy-paste and run it manually to reproduce the hang. Current behavior: timeout error says "command timed out or was cancelled" with no command вЂ” admin has no way to debug without reading source code.

### Session 2026-02-27

- Q: Should the global cache/state directory use `~/.complytime/` (Go/Podman pattern), XDG Base Directory, or a hybrid? в†’ A: `~/.complytime/` (dot-prefixed, single directory in HOME). All global state вЂ” policies, providers, state.json вЂ” lives under one predictable location. Matches `.docker`, `.kube`, `.cargo` convention for tool-managed state that admins interact with via CLI, not by browsing directly. Dot-prefix keeps `$HOME` clean. - Q: Should the system-wide provider directory (`/usr/libexec/complytime/providers/`) be codified in the spec as a fallback for RPM-installed providers? в†’ A: Yes. Dual-directory discovery: user directory (`~/.complytime/providers/`) checked first, system directory (`/usr/libexec/complytime/providers/`) as fallback. User-installed providers take precedence over system-installed. Matches FHS conventions for package-managed executables. FR-029 updated to reflect dual-directory discovery (Session 2026-02-27).
- Q: Workspace-local `.complytime/` and global `~/.complytime/` share the same directory name вЂ” collision risk if workspace is `$HOME`? в†’ A: Accept the collision risk. Current behavior confirmed correct (Session 2026-02-27).

### Session 2026-02-23g

- Q: How should doctor discover which variables a scanning provider requires? в†’ A: Extend HealthCheck response with required variable fields. Providers declare the variable names they need; doctor validates those keys exist in the workspace config. This replaces the vague "global variables present if configured" check with concrete, provider-driven validation. Providers already implement HealthCheck and it runs during doctor вЂ” adding fields is low-friction. - Q: Should required variables in HealthCheck distinguish between global and target scopes? в†’ A: Yes. A single flat list would force doctor to guess which section a variable belongs to (Session 2026-02-23g).
- Q: Should `get` run before `doctor` so doctor always has full policy context for variable validation? в†’ A: Yes. Require `get` before `doctor`. ## User Scenarios & Testing *(mandatory)*

### User Story 1 - Initialize Workspace and Fetch Policies (Priority: P1)

A system administrator needs to set up complyctl in their environment by configuring workspace targets and fetching the latest Gemara policies from a centralized OCI registry. This enables the admin to establish a local cache of policies that can be used for scanning even when the OCI registry is unavailable.

**Why this priority**: This is the foundational workflow that must be completed before any scanning can occur. Without initialization and policy retrieval, the system cannot function.

**Independent Test**: Can be fully tested by running `complyctl init` and verifying that workspace configuration is created. Running `complyctl init` again errors with "`complytime.yaml already exists`" (like `go mod init`). Then run `complyctl get` to fetch policies and `complyctl doctor` to validate the environment. This delivers a working local policy cache for scanning.

**Acceptance Scenarios**:

1. **Given** a fresh installation with no `complytime.yaml`, **When** an admin runs `complyctl init`, **Then** the system prompts interactively for policy URLs (with optional shortname ID per policy) and targets (referencing policies by effective ID), creates `complytime.yaml`, and exits. No `get` or `doctor` вЂ” user runs those separately (Session 2026-02-25d)
1a. **Given** a workspace with an existing `complytime.yaml` (e.g., committed to repo for CI/CD), **When** an admin runs `complyctl init`, **Then** the system errors with "`complytime.yaml already exists`" (like `go mod init`). Admins with existing config run `complyctl doctor` and `complyctl get` directly
2. **Given** a configured workspace, **When** an admin runs `complyctl get`, **Then** the system fetches only new or modified Gemara objects (Layers 1-3) from the OCI registry and updates the local cache atomically
3. **Given** a workspace with existing cached policies, **When** an admin runs `complyctl get` and the OCI registry has no changes, **Then** no downloads occur and the local cache remains unchanged
4. **Given** a `complyctl get` operation in progress, **When** a network failure occurs mid-sync, **Then** the system rolls back to the previous known-good cache state and reports the error

A system administrator needs to translate a declarative Gemara policy into concrete implementation artifacts (via `generate`), optionally review them, and execute a compliance scan. **Why this priority**: This is the core execution workflow that delivers the primary value of compliance scanning. It depends on User Story 1 being complete but can function independently once policies are cached.

**Independent Test**: Can be fully tested by running `complyctl scan --policy-id <ID>` (auto-generates and scans in one step) or by running `complyctl generate --policy-id <ID>` followed by `complyctl scan --policy-id <ID>` (explicit two-phase). `generate` can be tested independently to verify execution plan output without scanning.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST use go-gemara as the primary library for parsing, validating, merging policies and running assessments
- **FR-002**: System MUST fetch Gemara content (Layers 1-3) from a centralized OCI registry using the OCI Distribution Specification. Policies MUST be stored as OCI artifacts with `GetDefinitions` and `DefinitionVersion` resolver methods. Policy IDs MUST directly map to OCI registry content paths (e.g., policy ID `nist-800-53-r5` в†’ content path `registry.example.com/nist-800-53-r5`). Authentication MUST use standard OCI registry authentication mechanisms (Docker config, credential helpers, token-based auth) via `oras.land/oras-credentials-go` for credential resolution and `oras.land/oras-go/v2` for registry operations. Zero custom auth code вЂ” credential discovery delegated entirely to `oras-credentials-go` (Session 2026-02-14d). Client-registry API contract MUST follow existing OCI Distribution Specification and ORAS specifications (no custom contract definition required).
- **FR-003**: System MUST support workspace initialization via `complyctl init`. The workspace config file `complytime.yaml` is a static convention in the workspace root (like `go.mod`), not configurable via flags. `init` MUST error if `complytime.yaml` already exists (like `go mod init` errors when `go.mod` exists). `init` is config-creation-only: prompt interactively for policy URLs (with optional shortname ID per policy) and targets, create the config file, then exit. No `get`, no `doctor` вЂ” user runs those separately when ready (Session 2026-02-25d, supersedes Session 2026-02-25c dual-mode config). CI/CD path: commit `complytime.yaml` to the repo, then run `complyctl get` and `complyctl doctor` directly вЂ” do not run `init`. The config structure uses `PolicyEntry` objects: each policy has a `url` (full OCI reference including registry, e.g., `registry.com/policies/nist-800-53-r5@v1.0`) and an optional `id` (user-chosen shortname; if omitted, auto-derived from the last URL path segment). Policies from different registries coexist in one config вЂ” each URL is self-contained. Targets reference policies by their effective ID (explicit `id` or derived shortname). No `pack` field, no separate `registry` section вЂ” superseded by per-policy URLs (Session 2026-02-25d). Config sections: `policies` (list of `PolicyEntry`), `variables` (top-level global variables for workspace-scoped config like scan output directory), and `targets` (each with `policies` list of effective IDs and per-target `variables` for runtime config like credentials and profile). No `parameters` section вЂ” parameter values use policy defaults (Session 2026-02-24c). Session 2026-02-24: `evaluator_config` under PolicyConfig replaced by top-level `variables` (global) and `targets[].variables` (per-target)
- **FR-023**: ~~System MUST automatically run `complyctl get` after `complyctl init` completes to pull content to cache~~ (superseded by FR-003 Session 2026-02-25d: `init` is config-creation-only; user runs `get` and `doctor` separately)
- **FR-004**: System MUST support incremental sync using OCI registry pattern (OCI Distribution Spec manifests and digests) with the following capabilities:
  - Perform hash-based diff against OCI registry manifest digests in `complyctl get` to download only new or modified Gemara objects (digests computed client-side by `oras.Copy()` вЂ” no dependency on server `Docker-Content-Digest` header; Session 2026-02-14e)
  - Use OCI registry tag resolution to retrieve current manifest digests for version comparison
  - Use OCI manifest digests and tag comparison against local state file (`~/.complytime/state.json`) for optimization
- **FR-021**: Each policy ID MUST map to a single OCI manifest that contains all Layers 1-3 (bundles, guidelines, controls, policies, assessments) as separate layers within a multi-layer OCI Image Manifest. Each layer MUST be identified by its media type in the layer descriptor (e.g., `application/vnd.gemara.catalog.v1+yaml`, `application/vnd.gemara.policy.v1+yaml`, `application/vnd.gemara.guidance.v1+yaml`). The manifest config MUST be `application/vnd.oci.empty.v1+json`. Retrieval uses standard OCI Distribution endpoints per FR-002. The policy loader extracts layer content by matching media type strings from the manifest's layer descriptors (Session 2026-02-14e)
- **FR-022**: System MUST retrieve specific policy versions via OCI registry `GetDefinitions` API using definition version (semver-style versioning). When policy ID is specified without version in workspace config, system MUST resolve to latest available version. When policy ID includes explicit version (e.g., `nist-800-53-r5@v1.2.3`), system MUST use that exact version
- **FR-005**: System MUST only pull content that is specified in the configuration file (similar to `go get` behavior)
- **FR-006**: System MUST ensure local cache updates are atomic using `oras-go/v2/content/oci` OCI Layout store and `oras.Copy()` for remote-to-local sync; any failure during sync MUST roll back to previous known-good state. The OCI Layout store provides content-addressable blob storage with built-in digest verification
- **FR-007**: System MUST support `complyctl generate --policy-id <ID>` as an explicit command that resolves the dependency graph, invokes plugins via Generate RPC, persists the generated artifacts, records the policy cache digest at generation time, and outputs a structured plain-text execution plan: intro line with policy ID, then one indented block per target-provider pair showing Provider, Requirements count, and Status, then a conclusion line. No lipgloss table вЂ” `fmt.Fprintf` with alignment only. `generate` is optional вЂ” `scan` auto-generates when needed but shows only the one-line pre-scan summary (FR-034), not the full plan (Session 2026-02-23b, simplified Session 2026-02-26e)
- **FR-024**: System MUST require `--policy-id` parameter in `complyctl scan`. `scan` MUST be smart about generation (Session 2026-02-23b): (1) no generated artifacts exist в†’ auto-generate (resolve graph, invoke Generate RPC, persist); (2) artifacts exist and cached policy digest matches generate-time digest в†’ reuse existing artifacts; (3) artifacts exist but digest differs (policy updated via `get`) в†’ warn and auto-regenerate. System MUST scan all targets configured in workspace config for that policy ID (no explicit target selection needed)
- **FR-008**: System MUST resolve complete dependency graph (Controls, Guidelines, and Assessments) for the specified policy ID
- **FR-009**: System MUST communicate with scanning providers via local gRPC API (gRPC chosen over net/rpc for type-safe protobuf contracts, cross-language plugin support, and `buf`-based breaking-change detection; see Session 2026-02-14c). Three variable tiers flow through the scanning interface (Session 2026-02-24): **Generate RPC** MUST receive: (1) global variables вЂ” workspace-level configuration (e.g., scan output directory) defined once in workspace config, not per-target; (2) test variables вЂ” per-requirement parameters from the decomposed Gemara policy assessment plan (plan ID, requirement ID, and parameters map per requirement). Parameters use policy-defined defaults; no user-configurable parameter overrides exist (Session 2026-02-24c). Generate RPC is invoked during `complyctl generate` (explicit) or during `complyctl scan` (auto-generate when needed). Generated artifacts and the policy cache digest MUST be persisted so `scan` can detect freshness (Session 2026-02-23b). **Scan RPC** MUST receive only a list of targets (each with target ID and target variables map). No `requirement_ids` in ScanRequest вЂ” the scanning provider evaluates all requirements it was configured with during Generate (Session 2026-02-24). Scan RPC MUST return AssessmentLog for each requirement evaluated (containing steps run, results for each step, message, confidence level as `ConfidenceLevel` enum). ### Key Entities *(include if feature involves data)*

- **Workspace Configuration**: Maps targets (systems/environments) to policies via `PolicyEntry` objects (Session 2026-02-25d). Each policy has a `url` (full OCI reference including registry, e.g., `registry.com/policies/nist-800-53-r5@v1.0`) and an optional `id` (user-chosen shortname; auto-derived from last URL path segment if omitted). Policies from different registries coexist вЂ” each URL is self-contained. No `pack` field, no separate `registry` section. Targets reference policies by effective ID (explicit `id` or derived shortname). No parameter overrides вЂ” policy-defined defaults are used (Session 2026-02-24c). **Global variables** are workspace-level configuration (e.g., `workspace: ./.complytime/scan`) passed to scanning providers during Generate RPC. **Target variables** are per-target runtime configuration (e.g., `profile`, `kubeconfig`, credentials) passed to scanning providers during Scan RPC. **Test variables** are extracted from the decomposed Gemara policy assessment plan during policy resolution and passed during Generate RPC. Replaces old `evaluator_config` concept (Session 2026-02-24).
- **Gemara Policy**: Represents a complete policy definition including Layers 1-3. Has a unique policy ID that directly maps to an OCI registry repo name (e.g., `policies/cis-fedora-l1-workstation` в†’ `registry.example.com/policies/cis-fedora-l1-workstation`, flat mapping) and version information (semver-style). Each policy is a single multi-layer OCI manifest. Layers are identified by media type: `application/vnd.gemara.catalog.v1+yaml` (Layer 2 вЂ” Control Catalog: controls with assessment requirements and guideline-mappings), `application/vnd.gemara.guidance.v1+yaml` (Layer 1 вЂ” Guidance Catalog: guidelines referenced by control guideline-mappings), `application/vnd.gemara.policy.v1+yaml` (Layer 3 вЂ” Policy with assessment plans defining how to evaluate controls). Content is YAML. Manifest config is `application/vnd.oci.empty.v1+json`. For the OpenSCAP provider: XCCDF rules map to Gemara Controls, CIS item IDs (e.g., `cis-1.1.1.1`) map to guideline-mappings on those controls, and OVAL check definitions map to Policy assessment plans (Session 2026-02-24c).
- **Policy Cache**: Local storage of fetched Gemara objects using `oras-go/v2/content/oci` OCI Layout store, organized per policy ID under `~/.complytime/policies/{policy-id}/`. Each policy store follows OCI Image Layout (`index.json`, `oci-layout`, `blobs/sha256/`). Sync uses `oras.Copy()` from remote registry to local OCI store, providing built-in digest verification and content-addressable deduplication. Local state file (`~/.complytime/state.json`) tracks `last_sync` timestamp and per-policy metadata for incremental sync.
- **Dependency Graph**: Resolved set of Controls, Guidelines, and Assessments required for a specific policy ID. Resolved during `complyctl generate` (explicit) or during `complyctl scan` (auto-generate). Generated artifacts and the policy cache digest at generation time are persisted so `scan` can detect freshness and reuse artifacts when the policy hasn't changed (Session 2026-02-23b). Supports multiple evaluators вЂ” each assessment plan's `evaluation-methods[].executor.id` determines routing to the correct plugin.
- **Scan Result**: Output from executing assessment checks against targets. Contains AssessmentLog entries (one per requirement evaluated) with steps run, results for each step, message, and confidence level (`ConfidenceLevel` enum: not_set, undetermined, low, medium, high). Produces separate Gemara EvaluationLog file and formatted report (OSCAL/Markdown/SARIF).
- **Scanning Provider** (gRPC): External service that executes compliance checks. Identified by evaluator ID and communicates via local gRPC scanning interface. Providers are discovered automatically from `~/.complytime/providers/` directory by finding executable files matching naming convention `complyctl-provider-*` (not statically configured in workspace config). Evaluator ID is determined from executable filename (e.g., `complyctl-provider-kubernetes-evaluator` в†’ evaluator ID `kubernetes-evaluator`). No sidecar manifest files required вЂ” version and health verified at runtime via Describe RPC. Scanning interface consists of three RPC methods: Generate (receives global variables from workspace config + test variables from assessment plan per requirement), Scan (receives targets only вЂ” each with target ID and target variables; no requirement_ids вЂ” provider evaluates all requirements from Generate-time state), and Describe (verifies availability, reports version, and declares required variables). Target variables are owned by the system admin and contain per-target runtime configuration (auth tokens, connection strings, endpoints). Providers declare required variable *names* via `DescribeResponse.required_global_variables` and `required_target_variables` fields (Session 2026-02-23g, R51; renamed from HealthCheck вЂ” Phase 12); `complyctl doctor` validates these keys against workspace config. Documentation of valid *values* remains out-of-band. During `complyctl generate` (or auto-generate within `scan`), the system resolves the dependency graph and preps providers via Generate RPC with assessment configuration (formerly referred to as "OSCAL Policy"). During `complyctl scan`, the system executes assessment checks via Scan RPC with targets only, reusing generated artifacts when fresh. Providers return AssessmentLog entries containing steps run, results for each step, message, and confidence level (`ConfidenceLevel` enum). `generate` is explicit for AI/review workflows; `scan` auto-generates for simple cases (Session 2026-02-23b, Session 2026-02-24).

## Success Criteria *(mandatory)*
